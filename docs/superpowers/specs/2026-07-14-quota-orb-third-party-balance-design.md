# Quota Orb v1.1 第三方额度设计

## 1. 通俗流程

Quota Orb 每次刷新时按下面的顺序工作：

1. 读取当前生效的 Codex 配置，判断正在使用官方账号、CC Switch、Codex++，还是普通第三方 API。
2. 如果来源没有变化，继续沿用当前读取方式；如果来源变化，立即刷新一次。
3. 官方账号继续读取 5 小时和周额度百分比。
4. 已明确支持的第三方供应商读取余额；无法确认余额语义时不猜测，显示“需要配置”或“暂不支持”。
5. 请求失败时保留上一次成功结果，并标记为“数据已过期”，避免悬浮球瞬间变成完全不可用。
6. 所有 API Key 只在一次请求期间读取，不写入 Quota Orb 设置、日志或 UI。

## 2. 当前基线

本设计从 Quota Orb v1.0.1 开始实施。v1.0.1 已删除悬浮球中的 7D/2D 紧凑徽标，v1.1 不得重新引入这些控件或测试。

当前应用只有一个 `QuotaStateService` 和一个 15 秒刷新循环。v1.1 在这个结构上增量扩展，不新增第二套常驻调度系统。

## 3. v1.1 范围

第一阶段支持：

- 官方 Codex：保持现有百分比额度。
- DeepSeek：显示账户余额和币种。
- OpenRouter：显示当前推理 Key 的剩余额度上限，文案必须是“密钥剩余额度”，不能称为账户总余额。
- CC Switch 3.16.5 及兼容版本：只读识别当前 Codex 供应商，并复用其已选择的内置用量模板类型。
- Codex++：仅在能够确定单一普通 API 供应商及其凭据边界时支持。

第一阶段不支持：

- 执行 CC Switch 自定义 JavaScript 用量脚本。
- Codex++ 聚合供应商的单一余额。
- 根据 Relay 域名猜测真实扣费账户。
- 把密钥发送到与当前供应商不同的主机。
- 聚合多个供应商余额。
- 任意 POST 请求、重定向或通用脚本解释器。

未知供应商先显示“余额查询需要配置”。通用自定义 HTTPS 查询放到第二阶段，等真实供应商需求明确后再加入。

## 4. 统一显示模型

UI 使用一个简单的联合状态：

```text
UsageDisplayState
├─ OfficialQuota：百分比、重置时间、窗口名称
├─ MonetaryBalance：金额、币种、余额语义
├─ Unsupported：来源名称、原因、建议操作
└─ Error：错误摘要、上一次成功结果、是否已过期
```

`MonetaryBalance` 必须包含 `BalanceKind`：

- `AccountBalance`：例如 DeepSeek 账户余额。
- `ApiKeyLimitRemaining`：例如 OpenRouter 当前 Key 的剩余额度。

这样可以避免把不同供应商返回的数字都错误地标成“余额”。

## 5. 来源识别

新增轻量 `ActiveProviderDetector`，每次现有刷新循环开始时调用。

识别顺序：

1. 读取当前 Codex `config.toml` 中的 `model_provider` 和对应 provider 块。
2. 如果是官方配置，走现有 `CodexQuotaProvider`。
3. 如果 Base URL 是本机回环地址，且端口与 CC Switch 只读数据库中的 Codex 代理端口一致，再按 CC Switch 处理。
4. 如果配置明确来自 Codex++，读取其单一供应商状态；发现聚合、轮转或无法绑定真实供应商时返回 `Unsupported`。
5. 其他配置只按明确的已知主机映射到 DeepSeek 或 OpenRouter；未知主机不自动发出余额请求。

来源指纹只包含非敏感字段：来源类型、供应商 ID、规范化 Base URL 和余额模板类型。不得包含 Key、Token 或完整配置文本。

## 6. CC Switch 兼容策略

CC Switch 自身已经允许用户选择第三方余额模板。Quota Orb 不应重新猜测用户想查套餐、账户余额还是其他额度。

读取规则：

- SQLite 以只读模式打开。
- 只查询当前 Codex 供应商、`settings_config`、`meta` 和代理端口所需字段。
- 优先读取 `meta.usage_script.enabled` 与 `template_type`。
- 仅对白名单内置模板使用 Quota Orb 原生适配器。
- 自定义脚本、未知模板或已禁用查询均不执行，显示说明状态。
- 数据库字段缺失时失败关闭，不修改数据库。

兼容性按能力检测，不把 3.16.5 的完整表结构视为永久契约。至少用脱敏数据库夹具覆盖 3.16.5，并为较新结构保留失败关闭测试。

## 7. Codex++ 兼容策略

Codex++ 支持官方、混合 API、纯 API和聚合供应商，因此不能把 `CodexPlusPlus` 当成一个真实上游供应商。

规则：

- 只支持能够解析为一个普通 API 供应商的配置。
- 明确区分混合 API 与纯 API 的凭据位置。
- 聚合供应商、故障转移、按请求轮转和权重轮转返回 `Unsupported`。
- Relay Token 只能发送回配置明确声明的同一主机，不能转发给根据域名猜出的上游余额接口。
- 未知 Relay 进入“需要配置”，不尝试自动探测余额端点。

## 8. 余额适配器

### DeepSeek

- 请求：`GET /user/balance`
- 认证：当前供应商同主机的 Bearer Key。
- 返回：解析 `balance_infos`，保留币种；有多个币种时在详情中分项显示，不进行隐式汇率换算。
- 语义：`AccountBalance`。

### OpenRouter

- 请求：`GET https://openrouter.ai/api/v1/key`
- 认证：当前 OpenRouter 推理 Key。
- 返回：`data.limit_remaining`。
- 语义：`ApiKeyLimitRemaining`。
- `limit_remaining = null` 表示该 Key 没有配置额度上限，显示“密钥未设置额度上限”。
- 不调用需要 Management Key 的 `/api/v1/credits`。

## 9. 刷新与错误状态

继续使用现有 `QuotaStateService` 的一个循环和一个并发门：

- 每 15 秒检测来源指纹。
- 官方额度每轮读取，保持现有行为。
- 第三方余额至少间隔 60 秒读取一次。
- 来源变化或用户手动刷新时立即读取。
- 相同来源已有请求进行中时复用或跳过，不并发重复请求。

成功时保存最后成功结果和时间。失败时：

- 第一次且没有历史结果：显示错误。
- 有历史结果：继续显示旧值，同时显示“更新失败 / 数据已过期”。
- 切换到新供应商后不能显示上一个供应商的旧余额。

第一阶段不增加 5 秒独立计时器、后台请求调度器或复杂请求合并框架。如果 15 秒切换延迟经实际使用确认不可接受，再单独优化。

## 10. 安全边界

- 余额请求只允许 HTTPS，CC Switch 本机 SQLite 读取除外。
- 禁止重定向。
- 默认超时 10 秒，响应体上限 256 KB。
- 第一阶段只允许 GET。
- 请求主机必须与已确认供应商主机完全匹配。
- 不记录请求头、密钥、Token、完整供应商配置或原始响应。
- .NET 字符串不能保证内存擦除，因此不宣称 `SecretLease` 能彻底清除源字符串；重点是缩短作用域和阻止日志泄漏。

## 11. 设置与 UI

v1.1 第一阶段不增加通用自定义查询编辑器，因此 `AppSettings.SchemaVersion` 暂不需要升级。

悬浮球：

- 官方额度继续显示百分比。
- 金额使用简短格式，例如 `$12.34` 或 `¥28.00`。
- OpenRouter 显示“Key $12.34”或等价的明确文案。
- 无法查询时显示中性状态，不把“不支持”渲染成危险余额。

详情面板显示来源、供应商、数据语义、更新时间和过期状态，不显示密钥或内部配置路径。

## 12. 验证标准

至少覆盖：

- v1.0.1 的徽标删除结果未回归。
- 官方额度读取行为不变。
- 来源切换会清理旧供应商状态并立即刷新。
- DeepSeek 多币种解析。
- OpenRouter `limit_remaining`、`null` 和错误响应。
- CC Switch 3.16.5 只读识别、内置模板、自定义脚本和未知结构。
- Codex++ 单供应商、聚合供应商和未知 Relay。
- 超时、重定向、非 HTTPS、跨主机和超大响应被拒绝。
- 请求失败时保留同一来源的上次成功结果。
- 日志与设置中不存在测试密钥。

## 13. 后续阶段

只有第一阶段稳定后，才评估：

- 受限的同主机自定义 JSON 字段映射。
- 更多供应商原生适配器。
- 更快的配置变化监听。
- 多供应商余额聚合。

跨主机密钥发送、任意脚本执行和通用 HTTP 工作流不作为默认能力。
